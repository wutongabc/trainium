How the Endpoint Pool Works\n\n### planning_endpoint_pool.py\n\n**File**: `/dev3/zigeng/BrowseComp-Plus/search_agent/planning_endpoint_pool.py`\n\n**`PlanningEndpointPool`** is a thread-safe capacity-limited round-robin router:\n- Initialized with a list of ports and capacity per endpoint (can be heterogeneous via a mapping)\n- `acquire(preferred_port, affinity_timeout, allow_fallback)` -- leases one slot from the pool, optionally waiting for a preferred port with timeout before falling back\n- `release(port)` -- returns a slot\n- `endpoint(preferred_port, affinity_timeout)` -- context manager combining acquire/release\n- `assign_home()` -- returns a stable preferred endpoint without reserving capacity (used for trajectory affinity)\n- `retire(port)` -- permanently removes a dead endpoint\n- `begin_recovery(port)` / `restore(port)` -- temporarily removes an endpoint during restart\n\n**`PlanningEndpointRestartController`** coordinates with the Bash supervisor:\n- Communicates via filesystem: writes `tongyi-{port}.restart.{generation}` to trigger restart, reads `tongyi-{port}.generation` to detect recovery\n- Waits up to `timeout_seconds` for the supervisor to restart and declare a new generation\n- On success: calls `endpoint_pool.restore(port)`\n- On failure: calls `endpoint_pool.retire(port)`\n\n**`parse_planning_ports`** supports comma-separated lists with ranges (e.g., "6008-6014,6015").\n\n**`parse_planning_endpoint_capacities`** supports `port:capacity` override syntax.\n\n---\n\n## 4.
