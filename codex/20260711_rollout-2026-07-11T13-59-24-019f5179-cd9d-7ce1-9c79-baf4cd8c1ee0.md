## ASSISTANT (2026-07-12 12:32:44)

`128K bucket` 不是一个 128KB 文件，而是“为最大约 128,000 token 的静态输入形状编译一份专用程序”。

你的 Neuron 配置把不同上下文长度分成多个 bucket：

```text
Prefill/context encoding:
  新输入固定 32K token
  已缓存 prefix = 0 / 32K / 64K / 96K
  最大覆盖约 32K + 96K = 128K

Decode/token generation:
  16K / 32K / 64K / 96K / 128K
```

例如当前 KV 上下文有 70K token，decode 时会选择能容纳它的 96K bucket。Neuron 不会临时用任意 70K 形状执行，而是使用预先为 96K 静态形状编译的程序。

这带来至少 9 个逻辑图：

```text
4 个 prefill 图 + 5 个 decode 图
```

每个图都要完整经历：

```text
PyTorch/NxD graph
  -> HLO
  -> TP=4 分片
  -> 算子融合和布局
  -> SRAM/HBM 内存规划
  -> NKI/Neuron 指令生成
  -> NEFF 可执行文件
```

128K 图尤其重，因为 attention、KV cache、动态偏移、collective 和内存规划都要覆盖巨大的静态边界。更糟的是，当前 NxD 会同时编译这些图，最多启动 16 个 worker，导致主机内存突然耗尽。

NVIDIA vLLM 不容易遇到这个问题，主要是执行模型不同。

CUDA 内核通常支持动态尺寸。FlashAttention、GEMM、PagedAttention 等内核已经编译好，运行时把实际的序列长度、batch 和 block table 作为参数传进去：

```text
同一个 CUDA kernel
  -> 可以处理 8K、37K、70K、128K 等不同长度
```

NVIDIA vLLM 的 PagedAttention 也把 KV cache 切成页，调度器运行时组合这些页，不需要为每个上下文上限重新编译完整 Transformer。

Neuron 更接近：

```text
固定形状
  -> 编译器提前决定分片、内存地址、通信和执行计划
  -> 生成针对该形状的设备程序
```

CUDA 更接近：

```text
动态请求
  -> 调用一组已经编译好的通用 kernel
  -> 运行时调度不同长度和 batch
```
