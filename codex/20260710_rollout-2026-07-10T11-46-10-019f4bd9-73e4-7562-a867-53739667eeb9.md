因为 A100 和 Trainium 的执行模型不同。A100 主要是“加载权重后调用现成 CUDA kernel”，Trainium 主要是“先把整个模型编译成针对当前配置的静态程序”。

| 阶段 | A100 / CUDA | Trainium / Neuron |
|---|---|---|
| 权重格式 | vLLM 原生支持 HF 权重 | 需要适配 NxD 模型实现 |
| 算子执行 | cuBLAS、FlashAttention、Triton 等现成 kernel | 从 PyTorch/NxD trace 成 HLO |
| 动态 shape | 支持较好 | 更依赖固定 shape/bucket |
| 整图编译 | 通常不需要 | 必须运行 Neuron AOT 编译 |
| 最终产物 | CUDA kernel + CUDA Graph | NEFF 等 Trainium 可执行产物 |
| 配置变化 | 通常无需重新编译整个模型 | TP、序列长度等变化会产生新编译产物 |

**A100 实际也会转换一些权重**

例如：

- 按 tensor parallel 切分权重
- 合并 Q/K/V
- 转换 fused attention 或 fused MoE 所需布局
- 量化或反量化
- 把权重传到 GPU

但这些通常直接在内存中完成。vLLM 能直接读取 safetensors 并放到目标参数中，不需要先重新写一个约 120 GB 的本地模型。

CUDA 侧的大部分 kernel 已经存在：

- cuBLAS 提供矩阵乘法
- FlashAttention 提供注意力
- NCCL 提供多 GPU 通信
- vLLM/Triton 提供 paged attention、sampling、MoE kernel

A100 是 SM80，安装包通常已经包含相应二进制，或者只需对个别 Triton kernel 做小规模 JIT。它不需要根据整个 Qwen3-MoE 模型重新生成一份 GPU 可执行程序。

**Trainium 为什么必须编译**

Trainium 不直接执行 CUDA/PyTorch kernel。Neuron 需要提前知道：

- `TP_SIZE=4`
- `MAX_MODEL_LEN=8192`
- batch size
- bucketing 策略
- tensor shape
- MoE 分片方式
- NeuronCore 拓扑和通信方式

然后执行：

```text
PyTorch/NxD graph
  -> XLA trace
  -> HLO
  -> 算子融合、内存规划、分片和通信规划
  -> neuronx-cc
  -> Trainium 可执行产物
```

因此修改 TP、最大序列长度或 MoE kernel 后，原来的产物通常不能继续使用。

**这次为什么尤其慢**

除了 Trainium 本身需要 AOT 编译，当前 `vllm-neuron` 集成还有两个额外问题：

1. Tongyi 使用 Hugging Face Qwen3-MoE 格式，而执行端使用 NxD Qwen3-MoE 实现，需要额外转换权重布局。
2. 当前 loader 会删除默认的 hashed compiled-artifact 目录，导致重启时无法正常复用编译缓存。

所以这次慢并不完全是 Trainium 硬件本身导致的，而是：

```text

+ HF -> NxD 权重适配
+ 约 120 GB 本地写入
+ Neuron 整图 AOT 编译
+ 当前 loader 缓存处理有问题
```

A100 上主要只有“读取权重、传入显存、少量 kernel JIT/CUDA Graph warmup”，所以启动体验会明显更快。

它们分别控制模型如何分布到多个 NeuronCore，以及一次处理多少请求。

- `TP`，Tensor Parallel，张量并行  
  把同一层的权重和矩阵计算切到多个 NeuronCore 上。

  例如 `TP=4`：模型的每一层由 4 个 Core 共同计算。你的机器只有 4 个 Core，所以目前通常设置为 4。

- `CP`，Context Parallel，上下文并行  
  把长输入序列沿 token 维度切开，由多个 Core 分担 attention 的 prefill 计算。

  例如上下文为 128K、`CP=4`，概念上每个 CP 分片处理约 32K token，然后通过跨 Core 通信组合 attention 结果。

  CP 特别适合长上下文，但会增加通信，并且需要满足：

  ```text
  TP % CP == 0
  ```

  因此 `TP=4` 时，CP 可以是 `1`、`2` 或 `4`。`CP=1` 表示不使用上下文并行。

- `batch_size`，编译批大小  
  表示编译后的模型一次最多共同处理多少条序列/请求。在当前 vLLM 配置中，它通常来自 `--max-num-seqs`。

  ```text
  max-num-seqs=1  -> batch_size=1
  max-num-seqs=16 -> batch_size=16
  ```

  batch 越大，吞吐量通常越高，但 KV cache 和运行内存占用也越大，并且编译图可能更复杂。

```text

batch=1
context=128000
```

含义是：

```text

CP：拆长输入
batch：同时处理多少条输入
```

| 配置 | 含义 | 判断 |
|---|---|---|
| `TP=4, CP=1, batch=1` | 4 核拆模型，不拆上下文 | 当前配置，但 128K 压力最大 |
| `TP=4, CP=2, batch=1` | 两路上下文并行 | 可实验 |
| `TP=4, CP=4, batch=1` | 四路上下文并行 | 更适合长上下文，但通信更重 |
| `TP=4, CP=4, batch=16` | 官方思路的缩小版 | 128K 下很可能内存不足 |

关系可以简化为：

```text

```

例如一次请求先处理过：

```text
系统提示词 + 历史对话 = 8000 tokens
```

这些 token 的 K/V 已经缓存，那么：

```text
cached prefix = 8000
```

下一次增加 2000 个 token：

```text
当前新输入 = 2000
seqlen_k_total = 8000 + 2000 = 10000
```

### Cached prefix

Transformer attention 会为每个历史 token 计算 Key 和 Value。Prefix caching 将这些 K/V 保存起来。如果后续请求具有相同前缀，就不必重新计算。

例如：

```text

请求 B：共同的 10K 文档 + 问题 B
```

启用 prefix caching 后，请求 B 可以复用共同文档对应的 10K KV：

```text
cached prefix = 10000
```

它主要缩短首 token 延迟，不会扩大模型真实的 context window。

### seqlen_k_total

Attention 可以抽象为：

```text
Attention(Q, K, V)
```

当前新 token 产生 Query，同时需要关注：

```text
历史缓存的 K/V + 当前输入的 K/V
```

所以 NKI kernel 看到的总 K/V 序列长度就是：

## ASSISTANT (2026-07-11 05:07:22)

不是。关闭 `prefix caching` **不等于关闭 KV cache**。

这里有两层不同的缓存：

| 缓存 | 作用范围 | 关闭 prefix caching 后 |
|---|---|---|
| 普通 KV cache | 同一个请求内部，在逐 token 生成时复用历史 K/V | 仍然存在 |
| Prefix cache | 不同请求之间，复用完全相同的输入前缀 | 被关闭 |

普通推理流程仍然是：

```text
Prefill：计算整个 prompt 的 KV
Decode token 1：复用 prompt KV
Decode token 2：复用 prompt + token 1 KV
Decode token 3：继续复用之前所有 KV
```

所以生成每个新 token 时不会重新计算完整上下文，decode 性能不会因此崩掉。

关闭的是跨请求复用。例如连续发两个请求：

```text
请求 A：同一个 100K 文档 + 问题 A
请求 B：同一个 100K 文档 + 问题 B
```

启用 prefix caching 时，请求 B 可能复用请求 A 中相同的 100K 前缀，减少请求 B 的 prefill 时间。

关闭后，请求 B 需要重新计算这 100K prompt 的 KV，但进入生成阶段后仍会正常使用自己的 KV cache。

对 BrowseComp agent 来说，每一轮通常会把完整历史对话再次作为 prompt 发给服务端：

```text
第 1 轮：system + question
第 2 轮：system + question + tool call + tool result
第 3 轮：前面全部内容 + 新 tool result
```

如果 prefix caching 正常工作，后续轮次可以复用前一轮的大部分前缀。关闭后，每轮都要重新 prefill 完整历史，因此长轨迹的 TTFT 会明显变慢。影响主要是：

```text
每轮首次输出 token 更慢
```

而不是：

```text
每生成一个 token 都重新计算完整上下文
```

不过当前 NxDI 的 128K prefix bucket 组合无法通过 NKI 编译，因此现实选择是：

1. 保留 128K，关闭 prefix caching  
   能编译，后续 agent 轮次 prefill 较慢。

2. 降低编译上下文并保留 prefix caching  
   例如 context 设为 `126000`。但由于 bucket 中还可能出现 `65536 + 126000`，仅略微降低长度不能解决笛卡尔积越界；需要同时显式限制 active/prefix buckets。

3. 修改 bucket 配置，使每个组合满足：

   ```text
   active_bucket + prefix_bucket <= 131072
   ```

   但当前实现会对两个列表做笛卡尔积，因此必须限制最大值。例如：

   ```text
   active 最大 65536
   prefix 最大 65536
   ```

   这会限制单次新增 prompt 或可复用前缀的大小，而且需要验证 vLLM/NxDI 请求调度是否匹配这些 bucket。

现阶段最稳妥的是先关闭 prefix caching，成功编译并验证基本推理。之后再单独做一套受控 bucket 配置测试 prefix caching。编译产物缓存仍然保留，所以不会每次启动重新编译。

笛卡尔积就是：从两个集合中各取一个元素，生成所有可能的组合。
