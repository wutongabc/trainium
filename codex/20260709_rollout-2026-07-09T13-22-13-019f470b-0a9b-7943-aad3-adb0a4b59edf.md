## ASSISTANT (2026-07-09 15:34:13)

这三个问题其实是在拆迁移风险。A100 -> Trn2 不是“把 `CUDA_VISIBLE_DEVICES` 改成另一个变量”这么简单；Trn2 走 AWS Neuron runtime / compiler / NeuronCore，CUDA、flash-attn、NVIDIA vLLM Docker、`nvidia-smi` 这些假设都要重新处理。

我问这三个，是为了决定：我们是最小改仓库来跑通，还是重写成本更高但更原生地适配 Neuron。

**1. 目标：优先打通哪条路径？**

`vLLM Server`：
这是我推荐的路径。意思是 Trn2 上单独起一个 Neuron 版 OpenAI-compatible 推理服务，仓库里的 agent、judge、脚本都改成调用 `http://localhost:<port>/v1`。  
好处是改动最小：`search_agent/oss_client.py` 本来就是 OpenAI-compatible client 思路；bash 脚本也已经是“先起 vLLM server，再跑 client”。主要要改 `evaluate_run.py`，让 judge 不再直接 `from vllm import LLM`，而是也支持 OpenAI-compatible endpoint。  
风险是：具体模型必须被 Neuron vLLM/NxD 支持，或你要换模型。

`PyTorch Neuron`：
意思是把 Python 代码里的本地模型加载改成 `torch-neuronx` / NxD Inference 方式，直接在代码里控制编译、加载、推理。  
好处是更“原生”，将来可控性强。  
坏处是改动大：模型编译、静态 shape、KV cache、并行策略、batching 都可能碰。对这个仓库来说会影响 `evaluate_run.py`、本地 embedding/retriever、可能还有 agent 侧模型服务逻辑。除非你明确要深度适配 Trainium，否则不建议第一步走这个。

`只跑 API`：
意思是 Trn2 机器只跑检索、数据处理、缓存、评测调度，LLM 和 judge 都继续走 OpenAI/Anthropic/Gemini/远程 vLLM。  
好处是最稳，几乎不用碰 Neuron。  
坏处是 Trn2 算力没真正用起来。适合你现在只是临时把代码搬到 AWS 上，希望先保证实验能跑。

**2. 环境：AWS 机器基于什么安装？**

`Neuron DLAMI`：
推荐。AWS 预装/预配置 Neuron driver、runtime、工具链，少踩系统层坑。我们只需要设计 repo 的 `uv` 环境：哪些包保留，哪些 CUDA 包拆出去，哪些 Neuron 包用 AWS Neuron pip index 装。  
适合你现在这种“从 A100 环境迁过来，先跑通实验”的场景。

`Neuron DLC`：
也很推荐，尤其是如果你要跑 Neuron vLLM server。DLC 是 AWS 的 Deep Learning Container，推理服务和 repo uv 环境可以分开：容器管 Neuron/vLLM，repo 只当客户端。  
好处是隔离干净，不容易让 `uv.lock` 被 CUDA/Neuron 混合依赖搞乱。  
坏处是 bash 脚本要改成 Neuron 容器启动方式，调试多一层 Docker。

`手动 uv`：
意思是在普通 Ubuntu/EC2 上自己装 Neuron runtime、compiler、`torch-neuronx`、`neuronx-cc` 等。  
最灵活，但最容易版本错配。这个选项需要我们在计划里精确 pin AWS Neuron 版本、PyTorch 版本、Python 版本、pip extra index。  
除非你没有 DLAMI/DLC，或者要完全自定义镜像，否则不建议作为第一选择。

**3. 模型：现有模型名要保留到什么程度？**

`可换模型`：
推荐的最稳选择。比如现在仓库里有 Tongyi、Qwen、GPT-OSS、Qwen3 judge。Trn2 上不一定都能直接用 Neuron 高性能跑。这个选项表示：先用 Neuron 官方支持路径里容易跑通的模型替代，验证 pipeline，再慢慢补模型。  
优点是最快看到 Trn2 真正工作。

`尽量保留`：
这是折中方案。保留现有脚本接口和 run naming，能在 Neuron 上跑的模型就本地跑；暂时不支持的模型先走远程 OpenAI-compatible endpoint。  
我觉得这可能最适合你：仓库改动不会太大，实验名字和流程尽量不乱，同时避免被某个 unsupported 模型卡死。

`必须保留`：
表示 Tongyi/Qwen/GPT-OSS/Qwen3-32B judge 都必须在 Trn2 本地跑。  
这会把任务变成模型适配工程：要查每个模型架构是否被 Neuron vLLM/NxD 支持，是否要转 checkpoint，是否要编译，最大上下文、tensor parallel、batch size 怎么设。风险最高，计划也会更长。

我的默认建议是：**vLLM Server + Neuron DLAMI 或 DLC + 尽量保留模型**。也就是先把仓库改成“服务端推理后端可替换”，去掉 CUDA-only 假设；Trn2 负责能跑的 Neuron inference，不支持的模型先继续用远程 endpoint。

参考入口：AWS Neuron 文档里 Trn2/NxD/vLLM/torch-neuronx 都是独立路径，不能按 CUDA 方式直接套用：[PyTorch Neuron setup](https://awsdocs-neuron.readthedocs-hosted.com/en/latest/general/setup/torch-neuronx.html)、[Neuron Containers](https://awsdocs-neuron.readthedocs-hosted.com/en/latest/containers/index.html)、[Trn2 architecture](https://awsdocs-neuron.readthedocs-hosted.com/en/latest/about-neuron/arch/neuron-hardware/trn2-arch.html)。

`flash-attn` 是给 NVIDIA CUDA GPU 编译/运行的扩展，依赖 CUDA、cutlass/triton 等 GPU 栈。AWS Trn2 走的是 **Neuron runtime / Neuron compiler / NeuronCore**，不是 CUDA，所以：

- 不应该在 Trn2 默认 `uv sync` 里装 `flash-attn`
- 不应该跑 `uv pip install --no-build-isolation flash-attn`
- 也不应该依赖 `attn_implementation="flash_attention_2"` 这类 CUDA attention 路径

在 Trn2 上，高性能 attention 由 Neuron/NxD/vLLM-Neuron 自己的编译和 kernel 路径处理。也就是说，替代关系不是“装 Trainium 版 flash-attn”，而是“用 Neuron vLLM / NxD Inference 的 attention 实现”。

所以我把它移到：

```toml
[project.optional-dependencies]
cuda = [
  "deepspeed>=0.17.2",
  "flash-attn",
  "vllm>=0.9.0",
]
```

含义是：
- A100/CUDA：`uv sync --extra cuda`
- Trn2/Neuron：`uv sync`，然后用 DLAMI 里的 Neuron vLLM 环境起 server

这个方向是对的。

## ASSISTANT (2026-07-10 05:03:26)

`TP_SIZE` = **Tensor Parallel Size**，也就是把一个大模型的张量/权重切到多少个加速器单元上并行跑。

在你的脚本里：

```bash
TP_SIZE="${TP_SIZE:-32}"
```

然后传给 vLLM Neuron：

```bash
--tensor-parallel-size "$TP_SIZE"
```

含义是：这个模型用 **32 路 tensor parallel** 来加载和推理。

在 A100 上，`tensor_parallel_size=4` 通常表示用 4 张 GPU。  
在 AWS Trainium/Trn2 上，`TP_SIZE` 通常对应使用多少个 **NeuronCore / Neuron device partition**，具体可用数量取决于实例型号和 Neuron runtime 暴露的资源。

怎么选：
- 太小：模型可能放不下，或吞吐低。
- 太大：占用更多 NeuronCore，编译更慢，也可能浪费资源。
- 大模型一般需要更大的 `TP_SIZE`。
- 同一机器上同时跑 Tongyi 和 judge 时，要避免两边都默认占满资源。

当前脚本默认：
- Tongyi: `TP_SIZE=32`
- Qwen judge: `JUDGE_TP_SIZE=32`
- GPT-OSS 120B: `TP_SIZE=64`

这些是保守占资源的默认值，真正最优值要在 Trn2 上按模型能否编译、显存/NeuronCore占用和吞吐测试调整。
