## ASSISTANT (2026-07-12 07:23:36)

结论：慢的主因不是缺一个 `-j`，而是当前在做 30B MoE、128k 上下文、APC 多 bucket 的 NxD 静态图追踪与编译。当前任务已运行约 81 分钟，宿主有 12 个逻辑 CPU，但 `VLLM::EngineCore` 实际只持续吃一个核心；没有运行中的 `neuronx-cc` 子进程。已安装编译器的 `--help` 也没有暴露线程数参数。

最重的配置在 [serve_tongyi.bash](/dev3/zigeng/BrowseComp-Plus/bash/neuron/serve_tongyi.bash:7)：128k、并发 2、APC，编译 4 个 context-encoding 组合和 5 个 generation bucket。`TP_SIZE=4` 是 NeuronCore 张量并行，不是 CPU 编译并行。NxD 集成也同步调用 `self.model.compile(...)`，没有将这些任务拆进 worker 池，[实现见此](/dev3/zigeng/BrowseComp-Plus/.deps/vllm-neuron/vllm_neuron/worker/neuronx_distributed_model_loader.py:263)。

更糟的是缓存没有完成：两个 artifact 目录都只有 `neuron_config.json`，没有 `model.pt`，每次启动都会视为不完整并重新冷编译。当前脚本的 cache key 又严格包含 literal JSON、`EXTRA_VLLM_ARGS`、版本等，[见 launcher](/dev3/zigeng/BrowseComp-Plus/bash/neuron/serve_vllm_neuron.bash:77)，等价但写法不同的配置也会错过缓存。并且没有同配置编译锁，两个启动可能同时写同一目录。

优先改进：

- 让当前 production profile 完整跑出 `model.pt`，之后保持 `TP_SIZE`、长度、bucket、APC、cache tag 完全一致；可固定 `NEURON_COMPILED_ARTIFACTS` 指向该有效目录，避免实验重启触发编译。
- 分开 `dev/short` 和 `production-128k-apc` profile。大多数调试可用 `MAX_MODEL_LEN=32768`、`MAX_NUM_SEQS=1`、最少的 context/prefix/generation bucket 单独预编译。必须先按真实 prompt 长度验证，不能直接替换生产 128k。
- 给脚本补 `flock`、临时 artifact 目录加原子 promote、对 incomplete artifact 明确隔离；避免重复编译和坏缓存。
- cache key 对结构化 Neuron 配置做 canonical JSON，仅纳入真正影响图的参数；不要直接 hash 原始 `EXTRA_VLLM_ARGS`。
- 增加 trace、每个 bucket、NxD compile、load/serialize 的时间戳。目前日志无法判断到底哪一个 bucket 最慢。

不要直接尝试 `MAX_JOBS` 或 `neuron_parallel_compile`：前者不是当前 Neuron 路径的已验证开关；后者面向 PyTorch/XLA 训练脚本图提取，并未接入 vLLM/NxD。`neuronx-cc -O1` 虽然能减少 CLI compile 时间，但当前集成没有验证的透传接口，也不应盲塞。
