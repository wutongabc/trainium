**里面通常有什么**
`CODEX_HOME` 下面会有这些东西：

```bash
config.toml              # Codex 用户配置
auth.json                # 登录/认证信息，敏感
sessions/                # 普通对话记录，按日期分目录的 jsonl

plugins/                 # 插件
cache/                   # 缓存
log/                     # 日志
*.sqlite                 # app/状态/日志/记忆等数据库
```

**它不是工作目录**
`CODEX_HOME` 不等于你当前 repo，也不等于 `codex -C /path/to/repo` 的 `-C`。

- `CODEX_HOME`: Codex 自己的数据家目录。
- `cwd` / `-C`: Codex 当前要操作的项目目录。
- `AGENTS.md`: repo 内的项目级指令，跟着项目走。
- `$CODEX_HOME/config.toml`: 个人/全局 Codex 配置，跟着用户环境走。

**为什么要改它**
常见用途：

```bash
CODEX_HOME=/tmp/codex-clean codex
```

这样会启动一个“干净”的 Codex home，不会用你原来的登录态、配置、历史、插件等。

也可以长期设置：

```bash
export CODEX_HOME=/path/to/my-codex-home
codex
```

适合这些场景：

- 分离不同账号或不同实验环境。
- 临时测试 Codex 配置，不污染主环境。
- 把 Codex 状态放到更大的磁盘。
- 在恢复服务器或迁移机器时指定旧的 `.codex` 目录。

**怎么查看当前会用哪个目录**

```bash
echo "${CODEX_HOME:-$HOME/.codex}"
readlink -f "${CODEX_HOME:-$HOME/.codex}"
ls -la "${CODEX_HOME:-$HOME/.codex}"
```

**注意**
`CODEX_HOME` 里有敏感内容，尤其是 `auth.json`、对话记录、日志和可能出现的 API key/路径/私有代码片段。不要把整个 `.codex` 提交到 git，也不要随便打包分享。

```text
model_provider = micu
thread_source = user
```

所以它在 `-p micu` 下属于当前 provider，在默认 profile 下不属于当前 provider。

源码证据：

- `--profile micu` 会加载 `$CODEX_HOME/micu.config.toml`：`codex-rs/cli/src/main.rs` 的 `loader_overrides_for_profile()` 把 profile 转成 `user_config_path = $CODEX_HOME/<profile>.config.toml`。
- 官方 manual 也写了：`--profile profile-name` 会先加载 `~/.codex/config.toml`，再 overlay `~/.codex/profile-name.config.toml`。
- 配置解析源码里，`model_provider_id = model_provider.or(cfg.model_provider).unwrap_or_else(|| "openai")`。所以默认 profile 是 `openai`，`micu` profile 是 `micu`。
- `resume_picker.rs` 里 `picker_provider_filter()` 本地模式返回 `ProviderFilter::MatchDefault(config.model_provider_id)`。
- 同文件 `thread_list_params()` 把这个变成 `model_providers: Some(vec![default_provider])` 发给 session 列表后端。
- `--all` 只关闭 `cwd` 过滤。help 文档也只说 “disables cwd filtering”，源码里 `picker_cwd_filter(show_all)` 返回 `None`，但 provider filter 没被 `--all` 关闭。

所以精确结论是：

```text
codex resume -p micu
```

列的是：

```text
cwd 匹配规则 + model_provider = micu + 默认 source kinds
```

而：

```text
codex resume
```

列的是：

```text
cwd 匹配规则 + model_provider = openai + 默认 source kinds
```

即使 `cwd` 相同，`model_provider` 不同，picker 的候选集合也不同。因此同一个 `micu` session 不会出现在默认 profile 的 picker 里。

还有一个重要区别：如果你直接传 UUID，源码的 `resolve_session_thread_id()` 对 UUID 是直接解析，不走 picker 的 provider 过滤：

```bash
codex resume 019f4598-53be-7370-b714-92a67f5c3a64
```

如果这种直接 UUID 方式也失败，那就不是 picker 的 provider 过滤，而是后续用当前 profile/provider 接管旧线程时失败。要复现那一类，需要看具体报错。当前代码和配置已经能确定：**picker 不能跨 profile 看到同一 session，是因为 provider filter，不是 cwd。**
