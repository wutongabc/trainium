## ASSISTANT (2026-07-20 12:37:07)

这是 **Bedrock 每日 Token 配额（TPD）触顶**，不是余额不足。

你当前 `us-east-2` 账号的 Opus 4.6 配额比较低：

- 单区域 TPD：`1,296,000`
- `us.anthropic...` 跨区域调用翻倍后：约 **2,592,000 quota tokens/day**
- 该每日配额标记为不可自行调整

Opus 4.6 的配额消耗并不等于计费 Token。AWS 对 Opus 4.6 输出采用 **5 倍 burndown**：

```text
TPD 消耗 =
普通 Input
+ Cache Write
+ Output × 5
```

Cache Read 不计入 TPD。按刚才 Opus 4.6 的数据：

```text
普通输入        2,654
Cache Write 1,277,845
输出          146,334 × 5 = 731,670
--------------------------------
TPD 消耗约   2,012,169
```

当时理论上只剩约：

```text
2,592,000 - 2,012,169 = 579,831
```

而每次请求开始时，Bedrock 会先预扣：

```text
当前请求输入 + max_tokens
```

Max 模式通常会设置很大的 `max_tokens`，再加上新的 Cache Write，很容易让一次请求的预扣额度超过剩余 TPD，于是出现：

```text
429 Too many tokens per day
```

即使模型最终不一定生成那么多 Token，请求开始时也可能直接被拒绝。AWS 官方明确说明了 [TPD、5 倍 output burndown 和 max_tokens 预扣机制](https://docs.aws.amazon.com/bedrock/latest/userguide/quotas-token-burndown.html)。

可行处理方式：

- 等每日配额恢复后重试。
- 关闭 Max，使用默认或 High，降低输出和工具调用轮数。
- 降低请求的 `max_tokens`。
- 暂时切换 Sonnet 4.5/4.6，它使用独立且通常更大的每日配额。
- 如果数据跨区策略允许，可考虑 `global.anthropic...` 推理配置；它有独立的 Global TPD 配额，但你当前账户的 Global Opus 4.6 配额也只有 `2,592,000`。
- 该 TPD 配额在 Service Quotas 中显示 `Adjustable: False`，无法直接点击申请提高；需要联系 AWS Support，或者等待账号随着实际使用建立更高配额。

所以结论是：**Max 模式确实很可能促成这次 429；它既增加实际输出，又会通过较大的 `max_tokens` 预占剩余每日额度。**
