## ASSISTANT (2026-07-20 17:05:57)

第二个问题问的不是“要不要做分步 prefill”，而是：

> 分步 prefill 做好以后，它是否还必须和 Neuron vLLM 的其他高级功能同时使用，并且我们要为这些组合承担正确性保证？

这里的“首版”指第一次提交的实现/PR；“覆盖”指代码明确支持、测试验证、出问题算这个改动的 bug。没有被首版覆盖的功能不一定要永久删除，但可能需要暂时禁止与分步 prefill 同时开启，或者标成未验证。

先把主流程讲清楚：

1. 普通文本请求进入模型。
2. `prefill` 读取整段 prompt，并把每个 token 的 K/V 写进 KV cache。
3. prompt 处理完后进入 `decode/TKG`，每轮生成一个新 token。
4. 你要修的只是第 2 步：prompt 太长时，把一次 prefill 拆成多次 CTE。

几个名词分别是：

- `纯文本`：prompt 里只有文字 token，没有图片、音频等输入。
- `causal LM`：Llama、Qwen 这类“根据前文预测下一个 token”的普通生成模型。
- `prefix caching`：如果某段开头以前算过，就直接复用那段 KV；这也是本方案用来让第 2、3、4 个 chunk 读取前面 KV 的基础设施。
- `continuous batching`：服务运行时动态把不同请求放进或移出 batch。Neuron 当前自定义 scheduler 就在做这个。
- `Spec` / `speculative decoding`：投机解码。先由 draft/EAGLE 一次猜多个 token，再由主模型验证，用来加速生成。它主要发生在 prompt 完成以后，但会改变 token 计数和 model runner 的状态更新方式。
- `EAGLE`、`fused speculation`：两种 Neuron 已支持的投机解码实现。
- `LoRA`：运行时给基础模型挂一个小型微调适配器。
- `multimodal`：prompt 中包含图片等非文本内容；切 chunk 时还要考虑图片 token 和视觉编码结果的边界。
- `DI` / `disaggregated inference`：把 prefill 和 decode 放在不同机器或节点，KV 需要跨节点传输。
- `structured outputs`：要求输出严格符合 JSON、语法或 schema；它会影响采样阶段。

三个原选项的实际含义：

1. `纯文本非 Spec`

   只保证最核心场景：普通文字 prompt、prefix caching、Neuron continuous batching、普通逐 token decode。  
   例如你的 2048→4×512 场景完整支持。暂不承诺它与 EAGLE、图片输入、跨节点 prefill/decode 同时使用。

   优点是改动小、最容易验证正确；缺点是启用了高级功能的用户暂时不能使用这个修复，或只能使用但不纳入首版保证。

2. `同时兼容 Spec`

   除了核心场景，还必须保证 prompt 分四轮做完后，可以无缝进入 EAGLE/fused speculative decoding。

   难点在于当前 model runner 为 Spec 特意不用 scheduler 的 `num_computed_tokens`，而是从 prompt 长度和生成 token 数推导位置；分步 prefill 恰好要求在 prompt 未完成时相信 scheduler 的计数。因此实现要明确区分：

   - prompt 未完成：使用 scheduler 传来的 512、1024、1536；
   - prompt 已完成且进入 Spec：恢复现有的特殊位置推导。

   这可以做，但必须增加 Spec 回归测试和硬件验证，避免修好 prefill 后把 EAGLE 弄坏。

3. `全部现有特性`

   除了 Spec，还要同时保证 LoRA、图片模型、DI、structured outputs 等组合。

   这不是简单“多写几个测试”：multimodal 会碰到 chunk 与图片 token 的边界，DI 会碰到 KV 跨节点传输和“新请求其实位于 decode 节点”的特殊判断。因此首个补丁会明显变大，排查问题也更困难。

我的建议仍是选 `纯文本非 Spec`：先把你明确描述的崩溃闭环修好，同时在代码里保留以后扩展 Spec 的分支边界。若你们生产环境已经默认开启 EAGLE/fused speculation，则应选第二项。

## ASSISTANT (2026-07-20 17:10:10)

明白。方案里不会写死 `512`。

统一定义为：

- `C = 最大 CTE suffix bucket`
- 普通情况下，每个未完成的 prefill step 调度 `min(C, 剩余 prompt tokens)`
- 最后一轮可以小于 `C`，由已有 bucket selector 向上匹配到能容纳它的 suffix bucket
- 例如 `C=256` 就拆成 8 轮，`C=1024` 就拆成 2 轮
- prefix 不是另一个固定步长，而是每轮开始时的 `num_computed_tokens`：`0、C、2C……`

如果启用了 EAGLE，NxDI 的 CTE bucket 选择会额外预留一个 KV block，因此实际每轮的新 prompt token 上限应为 `C - block_size`；未启用 EAGLE 时才正好等于 `C`。这个差异会在配置推导中自动处理，不让用户手算。

对于你说的“suffix bucket 可以随意设置”，实现还必须在启动时验证两件事：

- suffix bucket 必须为正，并能容纳 EAGLE 需要的额外 block（若开启）；
- 最大 prefix bucket 必须能覆盖多轮累积到的最长 prefix，否则只是把“第一轮装不下”的崩溃推迟到后面的某一轮。

能从 `context_encoding_buckets` 读到时自动推导；直接加载预编译模型、scheduler 启动时看不到真实 bucket 时，则要求提供同一个上限，并在 model runner 加载模型后与真实 bucket 再核对。

# Prefix Caching 分步 CTE Prefill

## 概要

以 [vllm-neuron 0.5.3](https://github.com/vllm-project/vllm-neuron/tree/release-0.5.3)、vLLM 0.16.0、NxDI 0.10 为基线。

复用现有 prefix-caching CTE 2D bucket，不启用 NxDI tile-based chunked prefill，继续保持 `enable_chunked_prefill=False`。设 `C` 为最大 suffix bucket，普通请求按 `C` 分步执行；最后一轮可小于 `C`。

NxDI 现有代码已具备“读取 prefix KV + 计算新 suffix + 写入新 KV”的能力，预计无需修改 NxDI。

## 实现改动

- 配置与启动校验（`platform.py`）

  - prefix caching 开启时，从 `context_encoding_buckets` 推导 `C`，不硬编码 512。
  - 将 `C` 写入现有 `long_prefill_token_threshold`，借助 V1 scheduler 限制单轮 token 数；保持总 token budget 和 `enable_chunked_prefill=False` 不变。
  - 若启用 EAGLE，单轮有效新 token 上限为 `C - block_size`，给 CTE 的额外重计算 block 留空间。
  - 验证所有可达的 `(suffix_len, prefix_len)` 都有 2D bucket 可容纳；无效 bucket、prefix 容量不足或无 prefix caching 时启动即报错。
  - 现场编译时自动推导；预编译模型若启动前看不到真实 bucket，允许用现有 `--long-prefill-token-threshold` 提供上限，并在模型加载后与真实配置核对。配置过大时启动失败，不等到请求运行时崩溃。

- 调度状态机（`core/scheduler.py`）

  - 将“没有输出 token 且 `num_computed_tokens < num_prompt_tokens`”识别为未完成 prefill。
  - 未完成 prefill 的优先级高于新 prompt 和 decode；每步只暴露一个 CTE 请求，其余请求临时放回 holdback，避免 CTE 和 TKG 混在同一批次。
  - 首轮之后继续调用 V1 scheduler，让其负责 KV block 分配和 `num_computed_tokens` 的 `0 → C → 2C…` 更新。
  - 当前请求完成全部 prefill 后再接纳下一个新 prompt，避免连续到达的新请求造成多个半完成 prefill 或饥饿。

- Model runner（`worker/neuronx_distributed_model_runner.py`）

  - 新请求和 cached context 请求都使用：
    - `start = num_computed_tokens`
    - `end = start + num_scheduled_tokens`
    - `full_context_lens = end`
    - `computed_context_lens = start`
  - CTE 输入保留 `prompt[:end]` 和对应 position IDs，使 NxDI 可以利用 suffix bucket 的空位重算少量 prefix；slot mapping 只允许新 suffix 写 KV。
  - cached 请求若仍处于 context phase，走 CTE 分支，不读取不存在的 `output_token_ids[-1]`；只有 prompt 完成后才走现有 TKG 分支。
  - `_update_states` 在未完成 prefill 时相信 scheduler 的 `num_computed_tokens`；进入普通或 speculative decode 后恢复现有位置推导，避免破坏 EAGLE/fused speculation。
  - 每轮设置 `prefill_completion_state`。中间轮强制返回空 sampled tokens，最后一轮才允许采样首个输出 token，随后进入 TKG。
  - 增加不变量检查：单次 continuous-batching 执行不得混合 CTE 和 TKG，并记录每轮 `start/end/suffix_limit/completed` 调试信息。

## 接口行为

- 不新增必填 CLI 参数。
- bucket 配置可任意设置；`512` 仅是示例。
- 配置可见时自动启用安全切分；预编译配置不可见时复用 `--long-prefill-token-threshold` 作为显式提示，并在加载后校验。
- 纯文本模型为首版支持范围。只有需要分步 prefill 时，多模态和 disaggregated inference 组合才明确拒绝；短请求的现有行为不变。
- 保留并回归验证 speculative decoding；未启用 EAGLE 时，每轮恰好处理 `C` 个新 token（最后一轮除外）。

## 测试与验收

- 单元测试

  - 参数化 `C=256/512/1024`，覆盖整除、非整除、短于 bucket、冷请求和已有 prefix hit。
  - 验证 2048、`C=512` 时调度序列为四次 CTE，`num_computed_tokens` 为 `0/512/1024/1536`，之后才出现 TKG。
  - 验证 partial prefill 会压住新 prompt 和 decode，但完成后队列顺序恢复。
  - 验证 cached context 不访问输出 token，block table、slot mapping、position IDs 和 context lengths 正确。
  - 验证中间轮 CPU/on-device sampling 都不产生 token，最终轮只产生一次首 token。
  - 覆盖普通 decode、EAGLE/fused speculation 状态切换及无效 bucket 的启动报错。
  - 保持当前 scheduler/model-runner 的 223 个基线测试全部通过，并运行完整 unit suite。

- Trainium 功能验证

  - 使用确定性采样，对比单次大 CTE 基线与分步 CTE 的首 token及后续生成结果。
  - 覆盖首次冷请求、warm prefix hit，以及填充 KV cache 后重新发送已被 eviction 的长 prompt。
  - 从日志确认每轮 suffix 不超过有效上限、prefix 单调增长、中间轮无输出，最终顺利进入 TKG。
  - 若实际服务开启 EAGLE，再运行一组 speculative decode 端到端用例。

- NEFF/NTFF 验证

  - 按 `neuron-nki-profiling` 流程启用 runtime inspection，保存 CTE/TKG 的 NEFF 和 NTFF。
  - 使用 `neuron-explorer capture --profile-nth-exec=2 --enable-dge-notifs` 跳过 warmup，并导出 `summary-json`。
  - 确认请求只命中预编译的 CTE 2D buckets，没有运行时重新编译；trace 顺序为多次 CTE 后一次 TKG。
  - 记录各轮延迟和 HBM 读写量，作为后续性能优化基线，但性能指标不替代 token 正确性验收。
